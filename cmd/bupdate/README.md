# bupdate

Reconstructs files from fidx (File Index) files by combining local and remote chunks.

## Overview

`bupdate` is a Go port of the C++ bupdate tool from the bup backup system. Given a remote fidx file and a directory full of existing files with their fidx indexes, it can reconstruct remote files by:

1. Reading chunk mappings from all local .fidx files
2. For each chunk in the target file:
   - Using the local chunk if available (with SHA verification)
   - Reading from the remote file if the chunk isn't available locally
3. Reconstructing the complete file efficiently

This version uses filesystem operations only (no HTTP) and reads remote files using regular open/seek/read calls.

## Usage

```bash
bupdate --local <local-dir> --remote <remote-dir> <target.fidx>
```

### Arguments

- `--local <dir>` (or `-l`): Local directory containing existing files and their .fidx indexes (default: current directory)
- `--remote <dir>` (or `-r`): Remote directory to read the target file and fidx from (required)
- `<target.fidx>`: The fidx file to process (e.g., `bigfile.bin.fidx`)

### Example

```bash
# Reconstruct bigfile.bin using chunks from ./cache and ./remote
bupdate --local ./cache --remote ./remote bigfile.bin.fidx
```

This will:
1. Load all .fidx files in `./cache` to build a chunk mapping
2. Read `./remote/bigfile.bin.fidx` to get the target file structure
3. Reconstruct `./cache/bigfile.bin` by combining:
   - Chunks from existing files in `./cache` (when available)
   - Chunks from `./remote/bigfile.bin` (when not available locally)

## FIDX File Format

FIDX files are generated by the `fidx-cmd.py` command from bup. The format is:

```
[Header: 8 bytes]
  - "FIDX" magic (4 bytes)
  - Version number (4 bytes, big-endian)

[Entries: N * 24 bytes]
  Each entry:
  - SHA-1 hash (20 bytes) - git blob hash of the chunk
  - Size (2 bytes, big-endian) - chunk size in bytes
  - Level (2 bytes, big-endian) - content-defined chunking level

[Footer: 20 bytes]
  - SHA-1 hash of everything above (verification checksum)
```

## How It Works

### 1. Loading Local Mappings

Scans the local directory for all `.fidx` files and builds an in-memory index mapping SHA-1 hashes to file locations:

```go
type FidxMapping struct {
    SHA      [20]byte  // Chunk's SHA-1 hash
    Filename string    // Which local file contains it
    Offset   int64     // Byte offset within that file
    Size     uint16    // Chunk size
}
```

The mappings are sorted by SHA for efficient binary search lookup.

### 2. Chunk Deduplication

For each chunk in the target file:
1. Look up the SHA-1 in the local mappings
2. If found locally:
   - Read the chunk from the local file
   - Verify the SHA-1 matches (as a git blob)
   - Use it if valid
3. If not found or verification fails:
   - Read the chunk from the remote file
   - Verify the SHA-1 matches
   - Write to output

### 3. Git Blob SHA-1

Chunks are hashed using git's blob format:

```
SHA-1("blob <size>\0<data>")
```

This matches the behavior of `git hash-object` and allows chunk-level deduplication across git repositories.

## Differences from C++ Version

This Go implementation:
- ✅ Removes HTTP functionality (uses filesystem only)
- ✅ Simplifies to single-file mode (processes one fidx at a time)
- ✅ Uses Go's standard library (no WvStreams dependencies)
- ✅ Maintains the same core algorithm and file format compatibility
- ✅ Verifies all chunks with git blob SHA-1
- ❌ No download queue optimization (simpler sequential processing)
- ❌ No progress callbacks (prints simple progress percentage)
