FIDX Parallelization Architecture Redesign
===========================================

Current State
-------------
The fidx tool is entirely single-threaded:
1. collectFiles() walks directories sequentially
2. Each file is processed one at a time
3. Chunking (the CPU-intensive part) happens serially
4. Output is written sequentially to a buffered writer

On a fast disk, the bottleneck is likely the rolling checksum computation
in FindSplitPoint(), which processes every byte to find content-defined
chunk boundaries.


Proposed Architecture: Pipeline with Worker Pools
-------------------------------------------------

The key insight is that file processing has three distinct phases that can
be parallelized independently:

  [File Discovery] -> [Chunking Workers] -> [Ordered Writer]


Stage 1: File Discovery (Single Goroutine)
------------------------------------------
Keep collectFiles() mostly as-is, but instead of returning a slice,
have it send filenames to a channel:

    fileChan := make(chan string, 1000)  // buffered for smoothness
    go func() {
        collectFilesToChannel(args, fileChan)
        close(fileChan)
    }()

This allows chunking to start immediately rather than waiting for the
entire directory tree to be walked.


Stage 2: Chunking Worker Pool (N Goroutines)
--------------------------------------------
This is where the parallelism pays off. Create a pool of workers that:

    type ChunkResult struct {
        Index    int              // preserve ordering
        Filename string
        FileSize uint64
        Mtime    uint64
        Entries  []FidxEntry
        IsRef    bool             // true if copied from reference
        Error    error
    }

    resultChan := make(chan ChunkResult, numWorkers*2)

    for i := 0; i < numWorkers; i++ {
        go chunkWorker(fileChan, resultChan, refMap)
    }

Each worker:
1. Receives a filename from fileChan
2. Stats the file
3. If unchanged from reference, packages existing entries (fast path)
4. Otherwise, opens file and runs ChunkFile()
5. Sends complete ChunkResult to resultChan

Workers are completely independent - no shared state except:
- Read-only access to the reference map
- Each has its own Rollsum instance

Number of workers: runtime.NumCPU() or a --jobs flag


Stage 3: Ordered Writer (Single Goroutine)
------------------------------------------
Results arrive out of order but must be written in order. Use a
reordering buffer:

    type orderedWriter struct {
        nextExpected int
        pending      map[int]ChunkResult  // results waiting to be written
        output       *bufio.Writer
        hash         hash.Hash
    }

    func (w *orderedWriter) receive(result ChunkResult) {
        if result.Index == w.nextExpected {
            w.write(result)
            w.nextExpected++
            // drain any buffered results that are now in order
            for {
                if r, ok := w.pending[w.nextExpected]; ok {
                    delete(w.pending, w.nextExpected)
                    w.write(r)
                    w.nextExpected++
                } else {
                    break
                }
            }
        } else {
            w.pending[result.Index] = result
        }
    }

This ensures output is byte-identical to the sequential version.


Assigning Indices for Ordering
------------------------------
The file discovery stage assigns monotonically increasing indices:

    index := 0
    for _, file := range files {
        fileChan <- FileWork{Index: index, Path: file}
        index++
    }

Workers preserve the index through processing.


Memory Considerations
---------------------
With N workers processing files in parallel, memory usage increases.
Mitigation strategies:

1. Limit pending buffer size - if orderedWriter.pending grows too large,
   apply backpressure by blocking result submission

2. Stream large file chunks - for very large files, consider breaking
   them into segments that can be written incrementally rather than
   buffering all FidxEntry results

3. Use a semaphore to limit concurrent large-file processing:

       largeFileSem := make(chan struct{}, 2)  // max 2 large files at once
       if fileSize > 100*1024*1024 {
           largeFileSem <- struct{}{}
           defer func() { <-largeFileSem }()
       }


Alternative: Per-File Parallelism for Large Files
-------------------------------------------------
For very large files (>100MB), a single file could dominate processing
time. Split the file into segments:

    segmentSize := 16 * 1024 * 1024  // 16MB segments

Each segment is processed by a different worker, but requires careful
handling at boundaries:
- Segment N must pass its trailing bytes (up to BLOB_MAX) to segment N+1
- Segment N+1's first chunk boundary includes those trailing bytes

This is more complex and may not be worth it unless files are huge.
Recommend implementing basic file-level parallelism first.


Reference Map Access
--------------------
The reference map (filename -> refFileInfo) is read-only during processing.
No synchronization needed - just pass it to each worker.


Error Handling
--------------
If a worker encounters an error (file disappeared, permission denied):
- Set result.Error and send to resultChan
- Writer goroutine decides: skip file, abort, or log and continue
- Current behavior skips unreadable files; preserve this


Progress Reporting
------------------
The progress struct needs minor updates:
- Use atomic counters for fileCount and totalBytes
- Workers call atomic.AddInt64(&p.totalBytes, chunkSize)
- Display goroutine periodically reads and reports


Graceful Shutdown
-----------------
On SIGINT:
1. Stop sending to fileChan
2. Let workers drain
3. Writer writes everything received so far
4. Clean up temp file


Implementation Order
--------------------
1. Add resultChan and orderedWriter (single worker first, verify correctness)
2. Increase to N workers
3. Add progress reporting with atomics
4. Add memory backpressure if needed
5. (Optional) Large-file segmentation


Expected Speedup
----------------
On an 8-core machine with fast NVMe:
- Current: 1 core maxed, others idle
- Proposed: Up to 8 cores doing chunking in parallel

Amdahl's law applies - some overhead for coordination, and I/O may become
the bottleneck on slower disks. But for the stated case (fast disk, chunking
is bottleneck), expect 4-7x speedup on 8 cores.


Code Structure Changes
----------------------
main.go changes:
- processMFIDX() becomes the coordinator: spawns workers, manages channels
- New: chunkWorker() function
- New: orderedWriter type and methods
- collectFiles() modified to write to channel

bupdate/chunking.go:
- No changes needed - ChunkFile() is already stateless per-file
- Rollsum is per-call, no global state


Thread Safety Audit
-------------------
Current code that must remain single-threaded:
- Directory walking (fine, that's stage 1)
- Writing to output file (fine, that's stage 3)
- Computing final hash (fine, accumulates in stage 3)

Code that's already thread-safe:
- ChunkFile() - all state is local
- os.Open/Read/Close - safe for different files
- Reference map reads - read-only after initialization
